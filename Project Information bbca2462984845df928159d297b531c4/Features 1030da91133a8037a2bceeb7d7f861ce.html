Documentation on features developed up until milestone 2

  • Alerts API Endpoint

    Feature Overview

    The Alerts API is designed to inform users about various campus safety alerts. It allows authorized personnel to create, fetch, and update alerts in real-time, notifying students and staff of emergencies or important safety information. Alerts include details like message content, status (active, closed), affected areas, and timestamps.

    Feature Specifications

    • API Base URL: /alerts

    Main Components:

    • Insert Alerts: Allows administrators to create new safety alerts.
    • Fetch Alerts: Retrieves all the current alerts.
    • Update Alert Status: Enables updating the status of an existing alert.

    Endpoints Documentation

    a. Insert New Alert

    • Method: POST
    • URL: /alerts/alerts
    • Description: Inserts a new alert into the database.
    • Parameters:
      • message: (String) The content of the alert.
      • status: (String) The current status of the alert (e.g., 'active', 'closed').
      • affected_area: (String) The geographical area affected by the alert.
      • timestamp: (Datetime) Optional. The time when the alert was created. Defaults to the current time if not provided.

    b. Fetch All Alerts

    • Method: GET
    • URL: /alerts/alerts
    • Description: Retrieves all the alerts from the database.
    • Response: A list of all the alerts, including message, status, affected area, and timestamp.

    Example Response:

    [
  {
    "id": 1,
    "message": "Fire at Convocation Dining Hall",
    "status": "active",
    "affected_area": "West Campus",
    "timestamp": "2024-09-16T12:00:00Z"
  },
  {
    "id": 2,
    "message": "Heavy Flooding at OLS Builing",
    "status": "closed",
    "affected_area": "East Campus",
    "timestamp": "2024-09-14T10:00:00Z"
  }
]

    c. Update Alert Status

    • Method: PUT
    • URL: /alerts/alerts/:id
    • Description: Updates the status of an existing alert based on its id.
    • Parameters:
      • status: (String) The new status of the alert (e.g., 'active', 'closed').

    Data Models

    • Table Name: safetyalerts
    • Fields:
      • id: (INT) Primary key, auto-incremented.
      • message: (VARCHAR) The content of the alert.
      • status: (VARCHAR) The status of the alert ('active', 'closed', etc.).
      • affected_area: (VARCHAR) The area impacted by the alert.
      • timestamp: (DATETIME) The date and time the alert was created.

    Security Considerations

    • SSL: The API supports SSL connections for secure communication. SSL configuration is controlled by the environment variable DB_SSL.
    • Authorization: Ensure only authorized users (e.g., campus security personnel) have permission to insert and update alerts.

    Environment Variables

    The API uses environment variables for database configuration:

    • MYSQL_HOST: The host for the MySQL database.
    • MYSQL_USER: The username for the MySQL database.
    • MYSQL_PASSWORD: The password for the MySQL database.
    • MYSQL_DATABASE: The name of the MySQL database.
    • DB_PORT: The port for MySQL.
    • DB_SSL: Whether SSL is enabled for the database connection.
  • Login API endpoint

    Feature overview

    The /login endpoint is used to authenticate users by validating their credentials (username and password). Upon receiving a POST request with the required user credentials, the endpoint checks if the provided username and password are valid. If the credentials are valid, a success response is returned. Otherwise, the endpoint responds with an appropriate error message.

    Endpoint Details

    • HTTP Method: POST
    • URL: /login
    • Authentication: No authentication required for accessing this endpoint, as it is used to obtain authentication by verifying user credentials.
    • Description: Validates user login credentials (username and password) against stored database values and returns a response based on the validity of the credentials.

    Request Format

    Request URL

    POST /login

    Request Headers

    • Content-Type: application/json â€“ The request body must be in JSON format.

    Request Body

    The body of the request should be a JSON object containing the following fields:

    FieldTypeRequiredDescription
    usernameStringYesThe username of the user attempting to log in.
    passwordStringYesThe password associated with the username.

    Sample Request Body

    {
  "username": "john_doe",
  "password": "mypassword123"
}

    Response Format

    The server returns a response based on the success or failure of the authentication attempt. The responses vary based on the result of the login validation.

    Successful Response

    When the provided username and password match the records in the database, the endpoint responds with a 200 HTTP status code and a message indicating successful login.

    • Status Code: 200 OK
    • Response Body:
      {
  "message": "Login successful"
}

    Unsuccessful Response - Invalid Credentials

    If the username or password is incorrect, the endpoint responds with a 401 HTTP status code, indicating the authentication failure due to invalid credentials.

    • Status Code: 401 Unauthorized
    • Response Body:
      {
  "message": "Invalid credentials"
}

    Error Response - Server Error

    If there is an error on the server side (e.g., issues with database access or unexpected exceptions), the endpoint responds with a 500 HTTP status code, and an error message describing the issue.

    • Status Code: 500 Internal Server Error
    • Response Body:
      {
  "error": "Server error: <error_message>"
}

    Example Error:

    {
  "error": "Server error: Cannot connect to the database"
}

    Feature Overview

    The /login endpoint performs the following actions:

    1. Extracting Credentials:
      • The endpoint expects the request body to contain the username and password.
      • These are extracted from the body for further processing.
    1. Login Validation:
      • The endpoint communicates with the LoginController to validate the user's login credentials by calling validateUserLogin(username, password).
      • The LoginController function checks the database for a matching username and password.
    1. Response Generation:
      • If the credentials are valid, the endpoint returns a 200 status code and a success message.
      • If the credentials are invalid, it returns a 401 status code and an error message.
      • If any server-related issue occurs (e.g., database connection failure), a 500 status code with a detailed error message is returned.

    Error Handling

    The endpoint is designed to handle potential errors gracefully:

    • Invalid Credentials: If the credentials are incorrect, the system responds with a clear 401 Unauthorized response.
    • Server Error: If an unexpected error occurs during the request processing (e.g., database failure), the endpoint returns a 500 Internal Server Error along with a descriptive message to help diagnose the issue.

    Database Schema

    The LoginController interacts with a database that contains two columns for user login information:

    1. Username (username): The unique identifier for each user.
    1. Password (password): The password associated with the username.

    Sample Table Schema:

    Column NameData TypeConstraints
    usernameVARCHARPRIMARY KEY
    passwordVARCHARNOT NULL

    Authentication Logic

    The LoginController uses the following logic to validate the user's credentials:

    1. The username and password are provided in the POST request.
    1. The LoginController queries the database to check if the provided username exists.
    1. If the username exists, the controller compares the stored password with the provided password.
    1. If both match, the user is considered authenticated, and a success message is returned.
    1. If they do not match, the system returns an unauthorized message.

    Possible Enhancements

    • Password Hashing: For security, it is recommended to hash the passwords in the database and compare hashed versions during login attempts.
    • Rate Limiting: To prevent brute-force attacks, consider adding rate-limiting middleware that limits the number of login attempts within a certain time frame.

    Testing the Endpoint

    • Ensure that the server is running and that the /login/login endpoint is accessible.
    • Use a tool like Postman or Curl to send a POST request with a valid/invalid username and password to verify the responses
  • Resources API endpoint

The routes are connected to a controller that interacts with the MySQL database to fetch the necessary data.

    • description: (String) A detailed description of the safety resource.
    • link: (String) A link providing more information or resources.
    • type: (String) General category of the safety resource (e.g., "fire incident").
    • subtype: (String, optional) A more specific category if available (e.g., "wildfire").

    Example Response (200):

    "title": "Safety Resource 1",
    "description": "This is a description of Safety Resource 1.",

    "type": "fire incident",
    "subtype": "wildfire"
  },
  {
    "id": 2,
    "title": "Safety Resource 2",
    "description": "This is a description of Safety Resource 2.",
    "link": "<https://www.wits.ac.za/safety/lightning-safety>",
    "type": "weather issue",
    "subtype": "lightning"
  }
]

    Example Error Response (500):

    {
  "message": "Error message describing what went wrong."
}

    Error Handling:

    • If any error occurs (e.g., database connection failure), the API responds with status 500 and an error message explaining what went wrong.

    2. GET /resources/safety-resources/:type

    Description:

    This route fetches all safety resources that match the specified type. For example, it can return resources related to medical emergencies, weather issues, etc.

    Request:

    • Method: GET
    • URL: /resources/safety-resources/:type
      • type: (String) The type of the safety resource to filter by (e.g., "fire incident", "medical emergency").

    Parameters:

    • type (required): The category of safety resources to be retrieved.

    Response:

    • Status Code: 200 OK if data retrieval is successful.
    • Status Code: 404 Not Found if no resources match the specified type.
    • Status Code: 500 Internal Server Error if there's an error retrieving the data.

    Response Body (200):

    A JSON array of safety resources filtered by type.

    Example Response (200):

    [
  {
    "id": 1,
    "title": "Safety Resource 1",
    "description": "This is a description of Safety Resource 1.",
    "link": "<https://www.wits.ac.za/safety/fire-prevention>",
    "type": "fire incident",
    "subtype": "building fire"
  },
  {
    "id": 2,
    "title": "Safety Resource 5",
    "description": "This is a description of Safety Resource 5.",

    "subtype": "wildfire"
  }

    Example Error Response (500):

    {
  "message": "Server error: Error message explaining what went wrong."
}

    Error Handling:

    • If the resource type is invalid or no matching resources are found, a 404 Not Found error is returned.
    • In case of server errors or database connection issues, the API responds with a 500 Internal Server Error.

    Features and Functionality:

    1. Fetching All Safety Resources:
      • Functionality: Users can fetch a list of all safety resources available in the system. The response includes a list of resources with attributes like title, description, link, type, and subtype.
      • Controller Method: getAllSafetyResources()
      • Route: /resources/safety-resources
    1. Fetching Safety Resources by Type:
      • Functionality: Users can filter safety resources by their type. The API will return all resources that match the requested type.
      • Types Supported:
        • Medical emergency
        • Weather issue
        • Fire incident
        • Natural disaster
        • Security threat
      • Controller Method: getSafetyResourcesByType(type)
      • Route: /resources/safety-resources/:type
    1. Error Handling:
      • Robust error handling is implemented to return user-friendly messages in case of any issues, such as database errors or invalid type filters.
      • The system logs errors to the server console, making debugging easier.
    1. Scalability:
      • New types and subtypes of resources can easily be added to the system without breaking existing functionality. For instance, additional types for medical emergencies, fire incidents, or natural disasters can be integrated into the database.

    Example Use Cases:

    • Campus Security Officers: Campus security can query safety resources by type (e.g., fire incidents) to provide quick links to relevant fire prevention or response protocols.
    • Students and Faculty: Can access safety resources for various emergencies, such as medical or weather issues, directly from the API and integrate them into front-end applications or mobile apps.

    Future Enhancements:

    • Pagination and Filtering: Implement pagination and advanced filtering for large datasets. For example, filter by subtype in addition to type, and paginate results to improve response time.
    • Caching: Implement caching mechanisms to reduce database load for frequently accessed resources.
    • Security: Add authentication and authorization mechanisms to ensure that sensitive safety resources are accessible only by authorized personnel.

  • Real-Time Push Notifications

    Feature Overview

    This documentation provides a comprehensive guide to implementing a notification system using Server-Sent Events (SSE) with a React frontend. It includes detailed instructions on integrating SSE into your React application and how to subscribe to the SSE endpoint for real-time notifications.

    End Users must have Notifications enabled on their device settings to receive notifications.

    Server-Sent Events (SSE) is a standard allowing servers to push updates to web clients over a single, long-lived HTTP connection. Unlike WebSockets, which provide full-duplex communication, SSE is designed for simpler, one-way communication from the server to the client

    Features

    • Real-time Notifications: Receive live updates from the server directly in the React application.
    • Automatic Reconnection: Built-in reconnection logic ensures that clients automatically reconnect if the connection is lost.
    • Simple Implementation: Uses standard HTTP and a text-based format for easy debugging and maintenance.
    • Unidirectional Communication: SSE allows the server to send updates to the client but does not support client-to-server communication beyond simple HTTP requests. This is ideal for scenarios where you only need to push data to clients, such as live notifications or real-time updates.
    • Built-in Reconnection: SSE automatically handles reconnections if the connection is lost. The browser will attempt to reconnect to the server with an exponential backoff strategy.
    • Text-Based Protocol: SSE uses a text-based format, making it easy to debug and work with

    Integration

    Other teams who want to receive live notifications from your system can follow these instructions:

    1. Subscribe to the SSE Endpoint
      • Endpoint: alerts/pushalerts
      • Purpose: Connect to the SSE stream to receive notifications.
      
const eventSource = new EventSource('{liveUrl}/alerts/pushalerts');
//include live URL
    1. Handle Incoming Notifications
      • Purpose: Process and display real-time notifications as they arrive.
      • Payload Example:
        
{
    "desc": "Sample incident description"
}
      • Code Example:
        
eventSource.onmessage = (event) => {
    const data = JSON.parse(event.data);
    // Process the notification data
    console.log('Notification received:', data.desc);
};
    1. Handle Errors
      • Purpose: Manage connection errors and provide fallbacks if needed.
      • Code Example:
        
eventSource.onerror = (error) => {
    console.error('EventSource failed:', error);
    eventSource.close();
};
    1. Close the Connection
      • Purpose: Clean up and close the connection when no longer needed.
      • Code Example:
        
eventSource.close();

    This React notification system using Server-Sent Events (SSE) enables real-time updates to be efficiently pushed to clients. By following the integration steps and using the provided payload format, other teams can easily connect to our notification system and receive timely information.
