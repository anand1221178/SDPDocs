Testing

  • Unit Tests: Developed unit tests using Jest to verify the functionality of API endpoints.
  • Integration Tests: Conducted integration tests to ensure the frontend and backend communicate correctly.

Folder and File Structure

Folder and File Structure

The folders will be split up according to our chosen architecture style:

Front end

Front end folders

Folder NameContents
pagesReact components that render the entire web page for a specific rout
componentsReact components to be used within web pages
stylesCSS files related to each component and page
iconsSVG icons
lottiesLottie (JSON) animation files
servicesServices that assist in middle-end linking frontend to backend

Front end file names

Components should be named with brief terms related to their purpose. Their related CSS file should have the exact same name (obviously with the .css extension instead of the .js extension). This will ensure that all files are visually related and easy to identify.

e.g. if a component is named “CustomButton.js” in the “/components” directory then its style should be “CustomButton.css” in the “/styles” directory

Commenting in code

Commenting in code

Header comments for all files

At the top of the code file, include the file name, who wrote it/contributed to it and a brief description of the file’s contents. Please follow the following format, filling in the necessary details. Note that the stars aligning the left hand are there for us to see that this is the beginning of the file and shows that this is the information for the file header and not for anything else.

/**
		*File: example_service.js
		*
		*Author: Mitchell
		*
		*Description:
		* Holds important functions that interact with the API
		* and will format the json objects needed for all listeners.
		*
		*/

Comments for functions/procedures

Above each function, include the name of the function, the arguments, its return item(s) and a brief description of the function. Follow the following format, filling in the necessary details:

/*
		Function: get_examples
		
		Description:
			Interacts with database to create an array of all examples
		
		Arguments:
			example_type - Types of examples to look for.
										 Should be "all", "weather" or "medical"
										 
		Returns:
			Array of Example objects
	*/

Comments for Models

Each model has its own attributes and functions. At the top of the file, just under the file header comments, please outline the model’s name, a description of the model, the structure of the model and a short description of each function or attribute. Please also outline how the object should be instantiated.

/*
		Model: Example
		
		Description:
			Example model holds information of records read from database
		
		Structure, functions and attributes:
			name - The name of the example
			date_created - The data the example was recorded
			list_of_stuff - The list of stuff the example holds
			get_age() - Get the age of the example in years
		
		Instantiation:
			new Example( <name as string> , <date_created as date> , <list_of_stuff as array> )
	*/
