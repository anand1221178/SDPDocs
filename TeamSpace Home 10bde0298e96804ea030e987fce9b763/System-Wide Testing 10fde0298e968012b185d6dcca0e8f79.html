System-Wide Testing

Campus Safety App

Over the past few weeks, we have been working on the implementation of the Campus Safety App, focusing on building robust and effective API endpoints, ensuring system readiness, and integrating essential features to enhance user experience. Below is a detailed summary of the testing process we have conducted to verify the functionality, performance, and stability of the app.

1. API Testing

We began by validating the various API endpoints, particularly those relating to incident reporting and user interactions. Our goal was to ensure that the API can efficiently handle different types of incidents, including natural disasters (fires, floods) and emergencies (riots). We tested the following:

  • Incident Endpoints: We simulated different incident types and verified the response, checking for correct data retrieval and appropriate status codes. We also stress-tested the endpoints to ensure they perform well under high traffic conditions.
  • User Endpoints: We focused on user authentication, profile management, and role-based access control. These were tested for security vulnerabilities and proper authorization checks.

Outcome: The API endpoints responded as expected, handling incidents efficiently and ensuring secure user interactions. Some minor optimizations were identified in response times, which were then improved.

2. UI/UX Testing

To ensure the app provides a smooth experience for users, we performed a thorough UI/UX test, focusing on both desktop and mobile platforms. Key areas tested include:

  • Responsive Design: We tested the app’s layout across multiple devices to verify that it is fully responsive, with consistent and intuitive interfaces.
  • Navigation: We checked the ease of navigating between different sections of the app, particularly during an emergency situation.
  • Incident Reporting Form: We tested the form submission for speed, accuracy, and user-friendliness, ensuring that all critical information is captured.

Outcome: The UI/UX was responsive and user-friendly, with minor tweaks made to improve clarity and accessibility. The incident reporting form is intuitive and effective in capturing data.

3. System Monitoring and Readiness Testing

We focused heavily on ensuring the app’s readiness for live deployment by implementing monitoring and backup mechanisms. This phase involved:

  • Monitoring: We tested our real-time monitoring setup using Azure Monitor to track the health and performance of the system. Alerts were configured to trigger in the event of downtime or performance degradation.
  • Backup and Recovery: Azure Backup was used to create automated backups of critical system components. We ran tests on the recovery process to ensure that data could be quickly restored if needed.

Outcome: Our monitoring and backup systems were successfully tested and performed as expected. Alerts were triggered during our simulated failure scenarios, allowing us to identify and resolve issues promptly.

5. Integration and Final Testing

Once individual components were verified, we performed a final integration test to confirm that all parts of the system work seamlessly together. This included:

  • Backend and Frontend Integration: We tested how well the Node.js backend communicated with the React frontend. Data was accurately retrieved and displayed.
  • Full Incident Workflow: We simulated the entire workflow from reporting an incident to receiving notifications and handling user responses.

Outcome: All components worked together smoothly, with no significant issues during the integration process.

// Command to login with given user details
Cypress.Commands.add('login', (student_number, password) => {
    cy.visit('/');
    cy.get('input.inputField[placeholder="Student Number"]').type(student_number); 
    cy.get('input.inputField[placeholder="Password"]').type(password);
    cy.get('button').click();
  });

// Comamnd to report each type of alert
Cypress.Commands.add('report', (reportTypes) => {
    cy.intercept('POST', '/alerts/alerts').as('postAlerts');
    reportTypes.forEach((type) => {
        cy.get('.navbar-report-button').click();
        cy.get(`.shown-navbar-button-${type}`).click();
        cy.get('.report-confirmation-button-yes').click();
        cy.wait('@postAlerts').its('response.statusCode').should('eq', 200);
    });

});

// Command to report each type of incident
Cypress.Commands.add('incident', () => {
    // cy.intercept('POST', '/incidents/report-incidents').as('postIncidents');
    cy.get('.navbar-report-button').click();
    cy.get(`.navbar-report-button`).click();
    cy.get('.detailed-report-type-item').each(($el) => {
        cy.wrap($el).find('.report-card-root').click();
        cy.get('.make-detailed-report-button-cancel').click();
    });
    cy.get('.detailed-report-close').click();
    // cy.wait('@postIncidents').its('response.statusCode').should('eq', 200);
});

// Command to check each tab in the navigation bar
Cypress.Commands.add('checkTabs', () => {
    cy.get('.menu-button-root').click();
